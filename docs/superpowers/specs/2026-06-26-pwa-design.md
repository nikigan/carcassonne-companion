# PWA Support — Design

**Date:** 2026-06-26
**Status:** Approved, pending implementation

## Goal

Make the Carcassonne Companion installable to the home screen and usable offline
for solo play, by turning the existing Vite + React SPA into a Progressive Web
App. Multiplayer behavior is unchanged — it inherently requires a connection.

## Decisions (locked)

- **Update strategy:** prompt-to-refresh. A localized toast with a *Refresh*
  button appears when a new version is available; the page never reloads on its
  own.
- **Install affordance:** browser-native only. No custom in-app install button.
  (Chrome shows its install icon; iOS Safari uses *Add to Home Screen* — iOS
  never fires an automatic prompt regardless.)
- **Icon:** reuse the current 🏰 emoji look (dark `#1f2937` rounded square + the
  castle emoji), consistent with the existing `favicon.svg`.

## Approach

Use **`vite-plugin-pwa`** in its **`generateSW` / Workbox** mode with
**`registerType: 'prompt'`**.

Alternatives considered and rejected:

- **Hand-rolled service worker + manifest** — reinvents precaching,
  cache-busting, and the update handshake. Fragile; not worth it.
- **`vite-plugin-pwa` with `injectManifest`** (custom SW source) — more control
  than needed. Solo offline support only requires precaching the app shell, so
  `generateSW` is the better fit.

## Components & changes

### Manifest (generated by the plugin)

The plugin generates `manifest.webmanifest` and injects the `<link>` into
`index.html`. Configured in `vite.config.ts`:

- `name: "Carcassonne Companion"`
- `short_name: "Carcassonne"` (≤ ~12 chars to avoid home-screen truncation)
- `description`: reuse the `package.json` description
- `display: "standalone"`, `start_url: "/"`, `scope: "/"`
- `theme_color: "#1f2937"` (already the `<meta name="theme-color">` value)
- `background_color: "#1f2937"` (matches the dark `color-scheme: dark`)
- `lang: "en"`
- `icons`: see below

Orientation is intentionally left unlocked (the app works in any orientation).

### Icons (reuse the 🏰 look)

Rasterize the emoji via a Chrome `<canvas>` (the real Apple Color Emoji font
renders a crisp 🏰; `sharp`/`resvg` cannot render color-emoji glyphs). Each is a
`#1f2937` square with the centered castle emoji. Committed to `/public`:

- `pwa-192x192.png` — purpose `any`
- `pwa-512x512.png` — purpose `any`
- `pwa-maskable-512x512.png` — purpose `maskable`; emoji inset into the inner
  ~80% safe zone so platform masks don't clip it
- `apple-touch-icon-180x180.png` — full-bleed dark square (iOS applies its own
  rounding)

`index.html` gains the `apple-touch-icon` `<link>` plus
`apple-mobile-web-app-capable` / `apple-mobile-web-app-status-bar-style` meta
tags. The existing `favicon.svg` link stays.

### Update prompt ("prompt to refresh")

- New `src/components/UpdatePrompt.tsx`, using the
  `virtual:pwa-register/react` `useRegisterSW()` hook.
- When `needRefresh` is true, render a small fixed bottom toast with a localized
  message and a *Refresh* button that calls `updateServiceWorker(true)`. Nothing
  reloads until the user taps it.
- Rendered in `App.tsx`, inside `LanguageProvider` so `useI18n()` works.
- New i18n keys in both `en` and `ru` (e.g. `updateAvailable` →
  «Доступно обновление», `refresh` → «Обновить»). TypeScript enforces
  completeness of the `Strings` table.
- Add `/// <reference types="vite-plugin-pwa/react" />` to `src/vite-env.d.ts`
  for the virtual-module types.

### Service worker / offline behavior

- Workbox **precaches all build assets**, so the app shell loads offline. Solo
  play already persists to `localStorage`, so it is fully usable offline.
- `navigateFallback: 'index.html'` (SPA deep links) with
  `navigateFallbackDenylist: [/^\/api\//]` so the SW never shadows the
  multiplayer WebSocket/API route. (Defense-in-depth: service workers don't
  intercept WebSocket upgrades anyway.)
- `cleanupOutdatedCaches: true`.
- No runtime caching strategies — there are no external GET APIs to cache, and
  multiplayer is a WebSocket (inherently online).
- SW disabled in `dev` (default). `devOptions.enabled` can be flipped on
  temporarily for local SW testing.

### Cloudflare interaction (primary risk)

`@cloudflare/vite-plugin` runs a multi-environment build (client + worker). The
implementation must confirm the PWA plugin emits `sw.js`,
`manifest.webmanifest`, and the icons into `dist/client`, and that they're
served as static assets. The existing `not_found_handling:
single-page-application` only affects *unknown* routes; real asset files are
served directly, so the SW and manifest resolve normally.

If plugin ordering or the double-build causes the SW/manifest to be missing from
`dist/client`, adjust plugin order/options until the build output is correct.

## Files touched

- `package.json` — add `vite-plugin-pwa` devDependency
- `vite.config.ts` — add `VitePWA({...})`
- `index.html` — `apple-touch-icon` link + `apple-mobile-web-app-*` meta
- `src/vite-env.d.ts` — PWA client type reference
- `src/i18n.ts` — `updateAvailable` / `refresh` strings (en + ru)
- `src/App.tsx` — render `<UpdatePrompt />`
- `src/components/UpdatePrompt.tsx` — new
- `public/` — four PNG icons

No changes to game/scoring/multiplayer logic.

## Testing / verification

No automated SW test (the existing `vitest` covers pure logic; a service worker
isn't meaningfully unit-testable here). Verification is manual:

1. `npm run build` stays green (the gate — type-check + build).
2. `npm run preview`, then in DevTools → Application:
   - manifest is valid and the app reports as installable,
   - the service worker registers and activates,
   - icons render at all listed sizes.
3. Toggle offline → reload → solo play (add players, score) works.
4. Trigger an update (rebuild + reload) → the *Refresh* toast appears and only
   reloads on tap.
5. iOS: *Add to Home Screen* shows the 🏰 icon and launches standalone.

## Out of scope

- Custom in-app install button / iOS install instructions UI.
- Runtime caching of any network data.
- Push notifications, background sync, or offline multiplayer.
